# pdfport.nvim → filetree.nvim: reusable patterns

**Purpose.** This is the audit called for in `FINISH.md`. pdfport.nvim is not itself a
filetree plugin, so this isn't a feature *port map* the way a Neo-tree-config audit would
be — it's the same kind of pass as the "Pass 2: github_stats.nvim" section of
`StefanBartl/filetree.nvim`'s own `docs/ROADMAP/NEOTREE_FEATURES.md` (a sibling repo, not
part of this checkout): architectural/UI patterns worth reusing, built and refined against
a real, smaller plugin with the same cross-platform/config-system/health-check concerns.

For each: **what** it is, **origin** (`file:line`), **thematic home** in filetree.nvim, and
**notes** on applicability.

| Feature | Origin | Thematic home | Notes |
|---|---|---|---|
| Idempotent `FileType` + augroup helper: `M.on_filetype(pattern, augroup_name, callback)` creates a named, `clear = true` augroup and binds one callback — re-running `setup()` clears and re-creates instead of accumulating duplicate autocmds/keymaps | `lua/pdfport/bindings/autocmds.lua:15-21` (`M.on_filetype`) | `util.tree_attach.on_attach` (the "opt-in future refactor" noted in filetree.nvim's own `docs/ROADMAP.md` under "Centralize FileType keymap binding") | Directly applicable: filetree.nvim currently has ~40 per-feature `FileType` autocmds and has explicitly deferred building a single dispatcher for this exact reason. pdfport.nvim already built the minimal version of that dispatcher (one pattern → one augroup → one callback) to fix a real bug (nvim-tree was registering its autocmd without an augroup, silently duplicating keymaps on repeated `setup()`). Worth reviewing before filetree.nvim builds its own from scratch. |
| Which-key registration guarded entirely by `pcall(require, "which-key")`, building a spec list only for the keymaps that are actually active (not disabled), then `pcall(wk.add, spec)` once | `lua/pdfport/bindings/keymaps.lua:47-58` (`M.register_which_key`) | `bindings/keymaps.lua` (filetree.nvim already has this) | Not a new idea — filetree.nvim's own `Zentral-Prinzipien.md` audit already marks this pattern implemented and battle-tested via `github_stats.nvim`. pdfport.nvim is a third independent implementation of the identical shape (guard → collect → single `pcall` register), which is corroborating evidence the pattern is sound, not novel work to import. |
| `false`-to-disable convention: every keymap action resolves via `M.resolve(opts)` where an explicit `false` suppresses that binding entirely (vs. `nil` falling back to the default), checked once at setup rather than scattered `if opts.x ~= false` checks | `lua/pdfport/bindings/keymaps.lua:35-43` (`M.resolve`) | `bindings/keymaps.lua` (filetree.nvim already has this, per its `Checklist.md` audit: "User-configurable keymaps, disable via `false` — ✅") | Cross-check, not a gap: same convention, independently arrived at. Confirms this is the right default rather than e.g. an empty-string sentinel (which `github_stats.nvim` uses instead — worth deciding which convention filetree.nvim standardizes on if it ever unifies keymap-resolution code across the author's plugins). |
| Cross-platform interpreter resolution: `M.python()` tries `python3`, then `python`, then `py`, caches the first hit; `has_python_module()` shells out via `vim.fn.system({python, "-c", "import "..module})` (table-form args, no shell string) and reads `vim.v.shell_error` — replaced a prior bug where `os.execute("python3 -c '...' 2>/dev/null")` broke on `cmd.exe` (single-quotes, `/dev/null`) | `lua/pdfport/platform/init.lua:62-84` (`M.python`, `M.has_python_module`) | `util/platform.lua`, `filetree/health.lua` | Same shape as the already-logged `github_stats.nvim` `command_exists` pattern (PowerShell/`command -v`/direct-exec fallback) — another independent confirmation that "resolve + cache the working command name, never build a shell string with OS-specific quoting" is the right primitive for any cross-platform tool/interpreter probe, including anything filetree.nvim shells out to (e.g. trash, git, external openers). |
| Shared "pick an open mode" picker: one module (`util/picker.lua`) owns the choices table and the `hover_select`-or-`vim.ui.select` fallback; all four tree-integration adapters (neo-tree, nvim-tree, netrw, oil) call the same `pick_and_open(path)` instead of each carrying its own copy (this session deduplicated 4 near-identical copies into 1) | `lua/pdfport/util/picker.lua` (whole module); public entry points `lua/pdfport/init.lua` `M.open()` / `M.extract()` / `M.register_backend()` | filetree.nvim's planned `pdf.backend = "pdfport"` preview dispatch (see filetree.nvim `docs/ROADMAP.md` → "Preview — granular per-type config") | This is the concrete integration point, not just a pattern: when filetree.nvim's preview feature dispatches a PDF node to pdfport.nvim, it should call `require("pdfport").open({ path = ..., mode = ... })` or `.extract({ path = ..., __callback = ... })` directly — both are stable, documented (`doc/pdfport.nvim.txt` §6) public entry points — rather than reimplementing a mode picker against pdfport internals. |
| `config/DEFAULTS.lua` returns a **function** (`return function() return {...} end`), called fresh by `config/init.lua`'s `M.setup()`/`M.get()` fallback, so nothing shares a mutable default table by reference | `lua/pdfport/config/DEFAULTS.lua`, `lua/pdfport/config/init.lua:14-19` | Coding convention (see `Arch&Coding.md` in this repo, and filetree.nvim's own `@types/` split note) | Small but easy to get wrong: a plain `M.DEFAULTS = {...}` table can get mutated in place by `vim.tbl_deep_extend("force", M.DEFAULTS, opts)` if the extend call ever targets it directly instead of a fresh copy. Returning a constructor function sidesteps the whole class of bug. Worth matching wherever filetree.nvim's own config defaults are held as a plain table today. |

### Not applicable

pdfport.nvim's domain-specific modules (PDF extraction backends, text/markdown rendering,
Claude/Ollama API clients) don't map to anything a filetree plugin needs and aren't listed
above.
